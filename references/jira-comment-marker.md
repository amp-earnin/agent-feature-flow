# Tracker comment marker

The `feature-brief-author` skill posts each generated brief as a comment on the source ticket. To avoid duplicates and confusion, every brief comment uses a fixed marker.

## The marker

The first line of the comment body MUST be:

```
🤖 Feature brief — generated by Claude Code feature workflow
```

This is how the workflow finds its own comments for idempotent updates.

## Full comment format

```
🤖 Feature brief — generated by Claude Code feature workflow
[click to expand]

<full markdown brief>
```

## Updates, not duplicates

If a `🤖 Feature brief` comment already exists on the ticket (e.g., the brief was regenerated after human-requested revisions), the skill updates the existing comment rather than posting a new one. Process:

1. Fetch the ticket's comments via the tracker's MCP.
2. Filter for comments whose body starts with the marker line.
3. If exactly one match → update it.
4. If no match → post a new one.
5. If more than one match → log a warning, update the most recent, leave the others alone.

## Do not touch human comments

The marker is unique to bot-generated content. The workflow never modifies or deletes comments authored by humans, even if they look similar.

## Customizing for non-JIRA trackers

The marker convention is tracker-agnostic. If your tracker is Linear, GitHub Issues, Shortcut, etc., use the same marker line — only the MCP tool that posts the comment changes.
