# Feature brief author

You take the distilled content produced by `gather-requirements` and persist it. Two outputs:

1. **Local file** at `.claude/features/<TICKET>/brief.md` — the canonical reference for all downstream stages.
2. **Tracker comment** on the source ticket — team-visible, durable record.

## Input

- `TICKET`: tracker ticket ID.
- `CONTENT`: the distilled markdown from `gather-requirements` (block of text starting with `TITLE:`).

## Steps

### 1. Apply the template

The template lives at `references/feature-brief.md` (relative to this skill's directory). Read it once.

Fill it by mapping the `CONTENT` sections (TITLE, SUMMARY, USER GOAL, ACCEPTANCE CRITERIA, etc.) into the corresponding template slots. Keep section headings exactly as in the template — downstream stages depend on them.

Add header metadata at the top:

```
<!-- generated-by: feature-brief-author -->
<!-- ticket: <TICKET> -->
<!-- generated-at: <ISO 8601 timestamp> -->
```

### 2. Write the local file

`mkdir -p .claude/features/<TICKET>` if needed, then write to `.claude/features/<TICKET>/brief.md`.

### 3. Post the tracker comment

Use your tracker's MCP "add comment" tool (JIRA: `mcp__claude_ai_Atlassian_Rovo__addCommentToJiraIssue`) with the full brief content prefixed by:

```
🤖 Feature brief — generated by Claude Code feature workflow
[click to expand]

<full markdown>
```

The `🤖 Feature brief` marker is how we identify our own comments for future updates. See `${CLAUDE_PLUGIN_ROOT}/references/jira-comment-marker.md` for the canonical marker contract. Do not delete or edit other comments.

### 4. Update state.json

Set `stages.brief.status = "complete"` and `stages.brief.asset = ".claude/features/<TICKET>/brief.md"` in `.claude/features/<TICKET>/state.json`. If the file doesn't exist, create it with the schema from `feature-flow-conductor` (or `${CLAUDE_PLUGIN_ROOT}/references/state-schema.json`).

## Output

Return to the caller: the local file path, the tracker comment ID, and a one-line summary.

## Constraints

- Idempotency: if a brief.md already exists and a `🤖 Feature brief` tracker comment exists, update both rather than creating duplicates. (Look up the comment ID via the tracker's "get issue" call → comments list, filter by marker.)
- Do not edit or remove tracker comments authored by humans.
- Do not synthesize new content — only template the input you were given.
